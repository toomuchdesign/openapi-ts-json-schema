# openapi-ts-json-schema

[![Build Status][ci-badge]][ci]
[![Npm version][npm-version-badge]][npm]
[![Coveralls][coveralls-badge]][coveralls]

Generate TypeScript JSON Schema modules (`.ts` files with `as const`) directly from your OpenAPI spec — one source of truth for runtime validation and static type inference.

## Why?

Your OpenAPI spec, runtime schemas, and TypeScript types drift when maintained separately. `openapi-ts-json-schema` keeps them in sync: it generates your OpenAPI definitions as TypeScript `as const` modules — plain [JSON Schema](https://json-schema.org/) objects that work with any JSON Schema-aware tool and can be passed to [json-schema-to-ts](https://github.com/ThomasAribart/json-schema-to-ts) for static type inference.

## Known Limitations

Check these before integrating:

- **External `$ref` duplication** — schemas shared via external `$ref`s are bundled inline and duplicated across output files; avoid for multi-file / monorepo setups
- **No `discriminator` support** — OpenAPI `discriminator` objects are silently dropped
- **Alias schema edge cases** — schemas that are a bare `$ref` string (e.g. `Foo: "#/components/schemas/Bar"`) have known edge cases with some `refHandling` combinations
- **No CLI** — generation requires a Node.js script
- **No multi-document merging** — each call processes a single OpenAPI document

## How this fits with `openapi-typescript`

[`openapi-typescript`](https://github.com/openapi-ts/openapi-typescript) and `openapi-ts-json-schema` serve different purposes and work well together.

- **Types only** — if all you need is a TypeScript type representation of your OpenAPI spec with no runtime involvement, reach for `openapi-typescript`.
- **Runtime + types** — if your application uses the schemas at runtime (validation, serialisation, schema registration, or any programmatic access), use `openapi-ts-json-schema` to emit `as const` modules and infer types from them via `json-schema-to-ts`. The output is plain JSON Schema, so it works with any JSON Schema-compatible tool.
- **Both together** — a valid approach: `openapi-ts-json-schema` drives the runtime path; for any schema where `json-schema-to-ts` inference fails (e.g. complex polymorphic shapes), you can fall back to the type generated by `openapi-typescript`.

## Installation

```
npm i openapi-ts-json-schema -D
```

## Quick Start

Create a Node.js script (e.g. `generate-schemas.mjs`) and run it whenever your OpenAPI document changes:

```ts
import { openapiToTsJsonSchema } from 'openapi-ts-json-schema';

const { outputPath } = await openapiToTsJsonSchema({
  openApiDocument: 'path/to/open-api-specs.yaml',
  targets: {
    collections: ['components.schemas'],
  },
});
```

Schemas are written to a folder mirroring your OpenAPI layout (default: `schemas-autogenerated/`).

## Example

From this OpenAPI definition:

```yaml
components:
  schemas:
    User:
      type: object
      properties:
        id: { type: string }
        name: { type: string }
      required: [id, name]
```

`openapi-ts-json-schema` generates:

```ts
// schemas-autogenerated/components/schemas/User.ts
export default {
  type: 'object',
  properties: {
    id: { type: 'string' },
    name: { type: 'string' },
  },
  required: ['id', 'name'],
} as const;
```

Use the generated schema for both runtime validation and static type inference:

```ts
import Ajv from 'ajv';
import type { FromSchema } from 'json-schema-to-ts';

import userSchema from './schemas-autogenerated/components/schemas/User';

const ajv = new Ajv();
const validate = ajv.compile<FromSchema<typeof userSchema>>(userSchema);

const data: unknown = {};
if (validate(data)) {
  // data is now typed as { id: string; name: string }
} else {
  console.error(validate.errors);
}
```

## CommonJS

`openapi-ts-json-schema` is ESM-only. Use a dynamic `import()` from CommonJS projects:

```ts
async function run() {
  const { openapiToTsJsonSchema } = await import('openapi-ts-json-schema');

  const { outputPath } = await openapiToTsJsonSchema({
    openApiDocument: 'path/to/open-api-specs.yaml',
    targets: {
      collections: ['components.schemas'],
    },
    moduleSystem: 'cjs',
  });
}

run();
```

## Options

### Core options

| Property                         | Type                                             | Description                                                                                                                                                                                                                                                                              | Default    |
| -------------------------------- | ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| **openApiDocument** _(required)_ | `string`                                         | Path to an OpenAPI document (JSON or YAML).                                                                                                                                                                                                                                              | —          |
| **targets** _(required)_         | `{ collections?: string[]; single?: string[]; }` | OpenAPI definition paths to generate JSON Schemas from (dot notation). `collections`: paths pointing to objects/records of definitions where each entry is generated (e.g. `["components.schemas"]`). `single`: paths pointing to individual definitions (e.g. `["paths./users/{id}"]`). | —          |
| **outputPath**                   | `string`                                         | Directory where generated schemas are written. Defaults to `schemas-autogenerated/` next to `openApiDocument`.                                                                                                                                                                           | —          |
| **refHandling**                  | `"import" \| "inline" \| "keep"`                 | How `$ref`s are resolved. See [`refHandling` detail](#refhandling-option) below.                                                                                                                                                                                                         | `"import"` |
| **moduleSystem**                 | `"cjs" \| "esm"`                                 | Controls how import specifiers are written in generated files. Match this to your consuming project.                                                                                                                                                                                     | `"esm"`    |
| **silent**                       | `boolean`                                        | Suppress log output.                                                                                                                                                                                                                                                                     | `false`    |

### Advanced options

| Property          | Type                                       | Description                                                                                         | Default |
| ----------------- | ------------------------------------------ | --------------------------------------------------------------------------------------------------- | ------- |
| **idMapper**      | `(params: { id: string }) => string`       | Rewrite internal schema ids before they are written as `$id` and `$ref` values in generated output. | —       |
| **schemaPatcher** | `(params: { schema: JSONSchema }) => void` | Called for every generated schema node; mutate the schema in place before output.                   | —       |
| **plugins**       | `ReturnType<Plugin>[]`                     | Plugins to extend or customise the generation process.                                              | —       |

### Full configuration example

```ts
import {
  fastifyIntegrationPlugin,
  generateSchemaWith$idPlugin,
  openapiToTsJsonSchema,
} from 'openapi-ts-json-schema';

await openapiToTsJsonSchema({
  openApiDocument: './openapi.yaml',
  targets: {
    collections: ['components.schemas'],
    single: ['paths./users/{id}'],
  },
  outputPath: './generated',
  refHandling: 'import',
  moduleSystem: 'esm',
  silent: false,

  idMapper: ({ id }) => id.toUpperCase(),
  schemaPatcher: ({ schema }) => {
    if (schema.properties && !schema.type) {
      schema.type = 'object';
    }
  },
  plugins: [generateSchemaWith$idPlugin()],
});
```

### `refHandling` option

Three strategies for how `$ref`s are resolved:

| Value      | Description                                                                                                                       |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `"import"` | Each referenced schema becomes a separate `.ts` file; cross-file TypeScript `import` statements are emitted. This is the default. |
| `"inline"` | Referenced schemas are inlined at every usage site (self-contained, no imports, but possible redundancy).                         |
| `"keep"`   | `$ref` strings are preserved as-is in the output — useful when you plan to resolve refs dynamically or via a plugin.              |

Circular references are handled per strategy:

- `"inline"` — circular refs are replaced with `{}`
- `"import"` — the JSON Schema is resolved, but TypeScript recursion halts (`any` type, TS error 7022)
- `"keep"` — circular refs are left unresolved

### `idMapper` option

Every schema receives an internal id derived from its path in the OpenAPI document (e.g. `/components/schemas/Pet`). `idMapper` lets you rewrite these ids before they appear as `$id` and `$ref` values in generated output.

**Strip the `/components/schemas/` prefix** so Fastify's `addSchema` / `getSchema` can look up schemas by a short name:

```ts
await openapiToTsJsonSchema({
  openApiDocument: './openapi.yaml',
  targets: { collections: ['components.schemas'] },
  refHandling: 'keep',
  idMapper: ({ id }) => {
    const prefix = '/components/schemas/';
    return id.startsWith(prefix) ? id.slice(prefix.length) : id;
    // '/components/schemas/Pet' → 'Pet'
  },
});
```

**Map to a flat, namespaced identifier** across multiple target collections:

```ts
idMapper: ({ id }) => id.replaceAll('/', '.').replace(/^\./, ''),
// '/components/schemas/Pet' → 'components.schemas.Pet'
// '/paths/~1users~1{id}/get' → 'paths.~1users~1{id}.get'
```

`openapiToTsJsonSchema` returns metadata alongside the generated files:

```ts
{
  outputPath: string;
  metaData: {
    schemas: Map<
      string,
      {
        id: string;
        $id: string;

        originalSchema: JSONSchema | string;

        shouldBeGenerated: boolean;
        fileContent?: string;
        absoluteDirName: string;
        absolutePath: string;
        absoluteImportPath: string;
      }
    >;
  }
}
```

## Plugins

Plugins let you generate extra artifacts from the same internal metadata, or mutate the generation process. Two plugins ship with the library:

### `generateSchemaWith$idPlugin`

Generates an additional named export exposing the schema with the [`$id` JSON Schema property](https://json-schema.org/draft/2020-12/draft-bhutton-json-schema-00#rfc.section.8.2.1).

```ts
import {
  generateSchemaWith$idPlugin,
  openapiToTsJsonSchema,
} from 'openapi-ts-json-schema';

await openapiToTsJsonSchema({
  openApiDocument: 'path/to/open-api-specs.yaml',
  targets: { collections: ['components.schemas'] },
  plugins: [generateSchemaWith$idPlugin()],
});
```

### `fastifyIntegrationPlugin`

Generates a `fastify-integration.ts` file that exposes:

- `RefSchemas` — a TS type that enables `json-schema-to-ts` to resolve `$ref` schema types
- `schemas` — an array of schemas ready to be registered with [`fastify.addSchema`](https://fastify.dev/docs/latest/Reference/Server/#addschema)

| Option           | Type                                                         | Description                     | Default                              |
| ---------------- | ------------------------------------------------------------ | ------------------------------- | ------------------------------------ |
| **schemaFilter** | `({ id, isRef }: { id: string; isRef: boolean }) => boolean` | Pick which schemas to register. | All `components.schemas` definitions |

```ts
import {
  fastifyIntegrationPlugin,
  openapiToTsJsonSchema,
} from 'openapi-ts-json-schema';

await openapiToTsJsonSchema({
  openApiDocument: 'path/to/open-api-specs.yaml',
  targets: { collections: ['components.schemas', 'paths'] },
  plugins: [
    fastifyIntegrationPlugin({
      schemaFilter: ({ id }) => id.startsWith('/components/schemas'),
    }),
  ],
});
```

Register the generated schemas with Fastify:

```ts
import { JsonSchemaToTsProvider } from '@fastify/type-provider-json-schema-to-ts';

import { RefSchemas, schemas } from './generated/schemas/fastify-integration';

const server =
  fastify.withTypeProvider<
    JsonSchemaToTsProvider<{ references: RefSchemas }>
  >();

schemas.forEach((schema) => server.addSchema(schema));
```

See [plugins documentation](./docs/plugins.md) for lifecycle details, the `fastifyIntegrationPlugin` Fastify-Swagger setup, and instructions for writing your own plugin.

### Plugin lifecycle

Plugins execute in array order through three sequential hooks:

1. **`onInit`** — after options are resolved, before the document is parsed
2. **`onBeforeGeneration`** — after schema metadata is built, before file contents are generated
3. **`onBeforeSaveFile`** — after file contents are generated, before they are written to disk

If a plugin throws in any hook, generation stops immediately and the error is re-thrown with the plugin name included:

```
[openapi-ts-json-schema] Plugin "myPlugin" failed during "onBeforeGeneration" hook
```

### Write your own plugin

```ts
import type { Plugin } from 'openapi-ts-json-schema';

const myPlugin: Plugin<{ optionOne: string }> = ({ optionOne }) => ({
  onInit: async ({ options }) => {
    /* validate or mutate options */
  },
  onBeforeGeneration: async ({ outputPath, metaData, options }) => {
    /* inspect or mutate metaData */
  },
  onBeforeSaveFile: async ({ outputPath, metaData, options }) => {
    /* post-process fileContent */
  },
});

export { myPlugin };
```

## How It Works

Given an OpenAPI document, `openapi-ts-json-schema`:

1. Resolves and dereferences `$ref`s via [@apidevtools/json-schema-ref-parser](https://github.com/APIDevTools/json-schema-ref-parser)
2. Converts OpenAPI schema objects to JSON Schema via [@openapi-contrib/openapi-schema-to-json-schema](https://github.com/openapi-contrib/openapi-schema-to-json-schema) and [`openapi-jsonschema-parameters`](https://www.npmjs.com/package/openapi-jsonschema-parameters)
3. Applies the chosen `refHandling` strategy (`import` / `inline` / `keep`)
4. Emits `.ts` files exporting each schema as `as const`, mirroring the OpenAPI document structure
5. Runs plugin hooks at each stage of the pipeline

## Troubleshooting

### Generated output is empty

Check that `targets` paths use dot notation and match your document structure exactly — `components.schemas`, not `components/schemas`. Inspect the `metaData.schemas` map returned by `openapiToTsJsonSchema` to see what was parsed.

### Types infer as `never` with `FromSchema`

Import the schema as a value, not a type: `import userSchema from './...'` (not `import type`). Then use `FromSchema<typeof userSchema>`. Using `import type` strips the runtime value and leaves `FromSchema` nothing to infer from.

### External `$ref` schemas are duplicated across output files

Known limitation — external refs are bundled inline by `@apidevtools/json-schema-ref-parser`. No workaround today. See [Known Limitations](#known-limitations).

### Plugins aren't running in the expected order

Plugins execute in array order through: `onInit` → `onBeforeGeneration` → `onBeforeSaveFile`. All plugins finish `onInit` before any `onBeforeGeneration` runs. Reorder the `plugins` array to control priority.
